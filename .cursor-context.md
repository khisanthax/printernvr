# Printer NVR - Cursor Context

## Project name
Printer NVR

## Project purpose
Printer NVR is a minimal web-based recording system for 3D printers.

Printers expose camera streams through go2rtc.

The app provides a lightweight web UI that lets the user:
- view multiple printer cameras
- start recording
- stop recording
- optionally set a duration so recording stops automatically
- manage cameras through a config-backed web UI
- browse, download, and manually delete local clips

Recording is performed by ffmpeg on a central host, not on the printers.

Printers only provide camera streams.

Recordings are written locally on the recorder host.

NAS syncing or archival is outside the MVP unless explicitly added later.

## Core architecture decisions
- Printers are stream sources only
- go2rtc is expected but Printer NVR should remain a generic stream consumer
- Printer NVR runs on a separate recorder host
- ffmpeg runs on the recorder host
- RTSP recording inputs should use TCP transport by default
- recording should target the primary video stream by default rather than copying all streams into MP4
- recordings are stored locally on the recorder host
- Docker-first deployment
- GitHub repository is the source of truth
- Keep architecture simple and maintainable

## Camera configuration model

Printer NVR supports two camera setup modes.

### go2rtc helper mode

The user may provide:
go2rtc_base_url  
stream_name (optional)

Example fields:

id: sv08_left  
name: SV08 Left  
go2rtc_base_url: http://sv08-left.local:1984  
stream_name: cam  
enabled: true  
output_subdir: sv08_left

The system may derive likely URLs:

preview → http://host:1984/stream.html?src=cam  
record → rtsp://host:8554/cam

This is a convenience helper only.

### manual URL mode

The user may provide explicit URLs:

preview_url  
record_url

Example fields:

id: sv08_left  
name: SV08 Left  
preview_url: http://sv08-left.local:1984/stream.html?src=cam  
record_url: rtsp://sv08-left.local:8554/cam  
enabled: true  
output_subdir: sv08_left

### Resolution rules

Manual URLs override generated URLs.

record_url must exist after resolution.

Preview resolution order:
1. manual preview_url
2. generated go2rtc preview URL
3. no preview configured

Do not fall back to RTSP record_url for browser preview.

## MVP scope

The MVP should include:

- dashboard with multiple camera cards
- camera preview
- recording start/stop
- timed recording
- local file storage
- optional recent clip visibility
- Docker deployment
- config-based setup

## Explicit non-goals for MVP

Do not add unless explicitly requested:

- NAS sync logic
- Synology integration
- database
- user accounts
- motion detection
- object detection
- Crowsnest integration
- cloud dependency
- network share browsing
- heavy frontend frameworks
- general CCTV features

## Technology preferences

- Python backend
- FastAPI preferred
- plain HTML/CSS/JS frontend
- Jinja templates acceptable
- ffmpeg subprocesses for recording
- JSON config files
- local filesystem storage
- Docker and Docker Compose

## Deployment expectations

Printer NVR runs in Docker.

Persistent data should be bind-mounted from the host:

config  
recordings  
logs  

The container should remain simple and single-purpose.

## Code style guidance

- keep code modular and readable
- prefer simple solutions
- avoid unnecessary abstractions
- avoid heavy dependencies
- separate config, state, recording, API, and UI concerns
- comment only where needed

## Known requirements

The app must start even if zero cameras exist.

If no cameras exist the dashboard should show an empty-state message.

Disabled cameras must not break startup.

RTSP streams must not be assumed to work directly in browser preview.

Recording and probe failures should expose useful ffmpeg/ffprobe diagnostics in runtime state and UI.
