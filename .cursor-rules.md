# Printer NVR - Cursor Rules

These rules define how code should be written and how architectural decisions should be made when working on Printer NVR.

## Core Philosophy

Printer NVR is a **small, practical, self-hosted tool**, not a large platform.

The goals are:

- reliability
- simplicity
- maintainability
- low operational overhead

Avoid complexity unless it is clearly necessary.

Prefer simple solutions that are easy to understand and maintain.

---

# Architecture Rules

## Backend

The backend stack is:

- Python
- FastAPI
- ffmpeg subprocess calls

Do not introduce additional backend frameworks.

Avoid adding:

- Django
- Flask extensions unless necessary
- Celery
- Redis
- message queues
- event buses

The application should remain a **single process web server**.

---

## Frontend

Frontend must remain **simple**.

Allowed:

- HTML
- CSS
- minimal JavaScript
- Jinja templates

Do not introduce:

- React
- Vue
- Angular
- Svelte
- heavy frontend build systems

The UI should remain lightweight and server-rendered.

---

## Storage

Recordings are stored on the **local filesystem**.

Do not introduce:

- databases
- object storage
- cloud storage

All storage should be handled through the filesystem.

External archival (NAS, rsync, etc.) is handled outside the application.

---

## Docker

Docker is the primary deployment method.

Requirements:

- single container
- ffmpeg installed inside container
- host bind mounts for:
  - config
  - recordings
  - logs

Do not introduce multi-container orchestration unless explicitly required.

Avoid Kubernetes complexity.

---

# Code Organization

Code should remain modular and predictable.

Recommended modules:

config
models
state
recorder
api
util

Responsibilities should remain separated.

Avoid mixing UI, recording logic, and configuration logic.

---

# ffmpeg Usage

Recording must be implemented using **ffmpeg subprocess calls**.

Rules:

- build command strings in a dedicated module
- for `rtsp://` inputs, prefer `-rtsp_transport tcp` unless there is a specific reason not to
- default printer-camera recording should target the primary video stream and avoid muxing unrelated streams into MP4
- capture stderr for debugging
- handle process lifecycle cleanly
- terminate processes gracefully when stopping recording

Do not introduce alternative recording frameworks.

---

# Configuration

Configuration should remain **file-based JSON**.

Supported configs:

- app config
- camera config

Do not introduce:

- YAML
- environment-only configs
- database configs

Configuration files should be easy for a user to edit manually.

---

# Error Handling

Always assume:

- cameras may disconnect
- streams may fail
- ffmpeg may crash

Errors must be:

- logged
- visible in runtime state

The application must not crash if one camera fails.

---

# UI Design Rules

The UI is a **tool interface**, not a product marketing site.

Requirements:

- simple layout
- camera cards
- clear status indicators
- minimal controls
- mobile-friendly layout

Avoid visual complexity.

---

# Git Workflow

This repository is already connected to GitHub.

Cursor is allowed to:

- create files
- modify files
- stage files
- commit
- push

After completing a meaningful feature or phase:

- create a commit
- push the commit

Use clear commit messages.

Examples:

Phase 0: project scaffold and Docker setup  
Phase 1: camera dashboard and status API  
Phase 2: ffmpeg recording engine  
Phase 3: recording controls UI  

Avoid giant commits.

Prefer smaller logical commits.

---

# Implementation Discipline

Follow the roadmap phases.

Do not implement future phases unless requested.

If a prompt requests Phase 2, do not implement Phase 4 features.

Stay within scope.

---

# When Making Architectural Decisions

Always prefer:

1. simpler design
2. fewer dependencies
3. easier deployment
4. clearer code

Avoid adding infrastructure that increases operational complexity.

---

# If a Feature is Unclear

Choose the simplest practical implementation and document the assumption.

Never block progress waiting for clarification if a reasonable choice exists.
